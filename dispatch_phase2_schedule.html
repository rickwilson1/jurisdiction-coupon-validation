Phase 2 — Delivery Scheduling

Adds Greg's delivery queue, Ofelia's schedule view, and a yard fulfillment log on top of Phase 1 Order Intake.

Scope — what this phase adds to Phase 1
Phase 1 — Order Intake live
Automated intake and notification. Delivery scheduling is manual.
  • Order detected and routed automatically
  • Customer gets correct email instructions
  • Greg gets an alert email per delivery order
  • Order written to Firestore
  • Greg schedules manually — phone call, no tool
  • No visibility into pending vs. scheduled deliveries
  • Ofelia asks Greg for updates
  • Greenery Log is a shared Excel file
Phase 2 — Delivery Scheduling this document
Adds scheduling tools, delivery visibility, and yard log.
  • Greg has a queue view — all pending deliveries in one place
  • Greg logs a confirmed date/time after calling customer
  • Firestore record updated with scheduled slot
  • Customer gets a "your delivery is scheduled" email automatically
  • Ofelia has a live schedule view — no more asking Greg
  • Yard staff log fulfillments via a simple browser form
  • Greenery log exportable as CSV/Excel for OCWR reporting
  • All data stays in Firestore — no new platform
What gets built — new components
Brian's Delivery Queue
Web app · Greg's daily tool
  • Lists all pending delivery orders new
  • Status: pending call → called → scheduled
  • One-click to open the schedule form
  • Enter confirmed date, time window, notes
  • Triggers customer confirmation email on save new
  • Simple URL — no login needed if X-API-Key in URL
Ofelia's Schedule View
Web app · read-only dashboard
  • Shows this week and next week's deliveries new
  • Pending (unscheduled) shown separately
  • Customer name, address, material, qty, date
  • Read-only — no edit access
  • Replaces the Friday email summary
  • Always live — reflects Greg's latest updates
Yard Fulfillment Log
Web app · yard gate tablet
  • Staff enter order number → fields auto-fill new
  • Confirm material type and qty loaded
  • Staff name / initials
  • Timestamp recorded automatically
  • Replaces shared Excel Greenery Log
  • Export to CSV/Excel for OCWR reporting new
New API Endpoints
Dispatch Service additions
  • POST /api/schedule-delivery new
  • Writes confirmed slot to Firestore order record
  • Triggers "delivery scheduled" email to customer
  • GET /api/delivery-queue — pending deliveries
  • POST /api/log-fulfillment — yard log entry
  • GET /api/fulfillment-report — CSV export
Customer — Scheduled Email
New automated email template
  • Sent when Greg confirms date/time new
  • "Your delivery is scheduled for [date]"
  • Includes time window and contact number
  • Replaces Brian having to email/call again
  • Ofelia CC'd as with all customer emails
Firestore — Extended Schema
order_events + fulfillments
  • order_events gains: scheduled_date, time_window, notes
  • order_events gains: status field (pending → scheduled → fulfilled)
  • New collection: fulfillment_log new
  • fulfillment_log: order_number, material, qty, staff, timestamp
  • All queryable — no schema migration needed

Greenery log — replacing shared Excel
What the yard log captures
  • Order number (links back to full order record)
  • Customer name (auto-filled)
  • Material type — compost or cover mulch
  • Quantity loaded in cubic yards
  • Yard location (Bowerman, Olinda, Prima Deshecha, etc.)
  • Staff initials
  • Timestamp (recorded automatically)
What it replaces and adds
  • Replaces shared Excel Greenery Log spreadsheet
  • Works on any tablet or phone at the gate
  • No file locking or version conflicts
  • All entries queryable in Firestore immediately
  • Export to CSV/Excel for OCWR compliance reporting
  • Agromin admin can pull reports by date range or yard
  • Prerequisite: confirm what OCWR actually requires in the log
Suggested build order
  1. 1
    Confirm Greenery Log requirements with OCWR prerequisite
    Find out exactly what fields OCWR requires and what format they want for reporting. This determines whether the yard log is a simple form or needs a specific export format.
  2. 2
    Add POST /api/schedule-delivery endpoint easy
    Updates the Firestore order record with confirmed date, time window, and notes. Triggers the "delivery scheduled" customer email. All the pieces are already in place from phase 1.
  3. 3
    Build Greg's delivery queue web page easy
    A simple HTML page served by the dispatch service. Lists pending deliveries from Firestore. Schedule button opens the schedule form. No framework needed — plain HTML + the existing API.
  4. 4
    Build Ofelia's schedule view easy
    Read-only page showing confirmed deliveries this week + next, plus a pending section. Queries Firestore directly. Replaces the Friday email summary from phase 1.
  5. 5
    Add yard fulfillment log form medium
    Browser form for yard staff. Order number auto-fills customer details from Firestore. Writes to a new fulfillment_log collection. Requires confirming device availability at each yard gate.
  6. 6
    Add CSV/Excel export for OCWR reporting medium
    GET endpoint that queries fulfillment_log by date range and returns a formatted CSV or Excel file. Format depends on what OCWR requires — confirm before building.
